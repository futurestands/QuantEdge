# QuantEdge Database Architecture (SaaS Standard)

## 1. Core Principles
- **Immutability**: Execution data (`trade_events`) and history are never deleted or overwritten.
- **Traceability**: Every record belongs to a `workspace` (Organization) and has an `audit_log` entry.
- **Polymorphism**: Generic `attachments` and `tags` support future content types.
- **Relational Integrity**: Strict Foreign Keys with appropriate `ON DELETE` actions to prevent orphaned data.

---

## 2. Entity Relationship Summary

### 2.1 Identity & Access
- `organizations` (Workspaces): The root owner of all data.
- `organization_members`: Maps users to organizations with roles (owner, analyst, viewer).

### 2.2 Infrastructure Layer
- `broker_connections`: Integration configurations (MT5, IBKR, etc.).
- `broker_accounts`: Specific accounts (e.g., FTMO $100k, IBKR Live).

### 2.3 Research & Logic
- `strategies`: Algorithm definitions.
- `strategy_versions`: Point-in-time logic used for specific trades or backtests.
- `research_projects`: Containers for strategic exploration.
- `research_versions`: Successive iterations of research.

### 2.4 Execution Layer (Discipline Guardian™)
- `trade_events`: The "Source of Truth" for an execution.
- `trade_event_history`: A ledger of all state changes (e.g., SL moved, partial close).
- `rule_violations`: Automated flags generated by the Discipline Engine.
- `discipline_profiles`: Configurable rulebooks.

### 2.5 Intelligence Layer
- `ai_reviews`: Generative analysis of trades or projects.
- `discipline_scores`: Temporal scores (daily, weekly, monthly).
- `optimization_jobs`: Computational tasks for parameter tuning.

## 3. Schema Definition (Migration Plan)

### Phase 01: Core & Access
- **`organizations`**: `id`, `name`, `slug`, `settings`, `created_at`.
- **`organization_members`**: `id`, `organization_id`, `user_id`, `role`, `joined_at`.
- **`tags`**: `id`, `organization_id`, `label`, `color`.

### Phase 02: Research & Strategies
- **`strategies`**: `id`, `organization_id`, `name`, `description`.
- **`strategy_versions`**: `id`, `strategy_id`, `rules` (JSONB), `parameters`, `version_label`.
- **`research_projects`**: `id`, `organization_id`, `strategy_id`, `name`, `status`.
- **`research_versions`**: `id`, `project_id`, `backtest_id`, `metrics`.

### Phase 03: Brokerage & Execution
- **`broker_connections`**: `id`, `organization_id`, `provider` (enum), `encrypted_credentials`.
- **`broker_accounts`**: `id`, `connection_id`, `account_number`, `type` (funded, demo, etc).
- **`trade_events`**: `id`, `broker_account_id`, `organization_id`, `strategy_version_id`, `symbol`, `direction`, `entry_price`, `exit_price`, `pnl`, `status`.
- **`trade_event_history`**: `id`, `trade_event_id`, `event_type` (sl_moved, closed, etc), `payload` (JSONB), `created_at`.

### Phase 04: Discipline & Guardian
- **`discipline_profiles`**: `id`, `organization_id`, `name`, `rules` (JSONB), `is_active`.
- **`rule_violations`**: `id`, `trade_event_id`, `discipline_profile_id`, `rule_key`, `severity`, `evidence`.
- **`discipline_scores`**: `id`, `organization_id`, `period` (daily, weekly), `score_execution`, `score_risk`, `score_psychology`.

### Phase 05: Journey & Media
- **`trade_notes`**: `id`, `trade_event_id`, `content`, `emotion`.
- **`attachments`**: `id`, `organization_id`, `resource_type`, `resource_id` (Polymorphic), `file_path`, `metadata`.
- **`trade_screenshots`**: Specialized view/table for quick lookup of entry/exit visuals.

## 3. Detailed Table Specifications

### Identity & Access
- **`organizations`**: 
    - PK: `id` (UUID)
    - Columns: `name`, `slug`, `created_at`
- **`organization_members`**: 
    - PK: `id` (UUID)
    - FK: `organization_id` -> `organizations`, `user_id` -> `auth.users`
- **`audit_logs`**:
    - PK: `id` (UUID)
    - FK: `organization_id`, `user_id`
    - Columns: `event_type`, `resource_id`, `old_value`, `new_value`

### Infrastructure
- **`broker_connections`**: 
    - PK: `id` (UUID)
    - FK: `organization_id`
    - Columns: `provider` (ENUM), `encrypted_credentials` (Vault), `status`
- **`broker_accounts`**:
    - PK: `id` (UUID)
    - FK: `connection_id` -> `broker_connections`, `organization_id`
    - Columns: `account_number`, `type`, `balance`, `currency`

### Execution (Discipline Guardian™)
- **`trade_events`**: 
    - PK: `id` (UUID)
    - FK: `organization_id`, `broker_account_id`, `strategy_version_id`, `research_project_id`
    - Columns: `external_id`, `symbol`, `direction`, `entry_price`, `exit_price`, `pnl`, `status`
- **`trade_event_history`**:
    - PK: `id` (UUID)
    - FK: `trade_event_id` -> `trade_events`
    - Columns: `event_type` (open, modify_sl, close), `payload` (JSONB), `occurred_at`
- **`rule_violations`**:
    - PK: `id` (UUID)
    - FK: `trade_event_id`, `discipline_profile_id`
    - Columns: `rule_key`, `severity`, `evidence` (JSONB), `is_resolved`

### Intelligence
- **`ai_reviews`**:
    - PK: `id` (UUID)
    - FK: `resource_id` (Polymorphic), `organization_id`
    - Columns: `model`, `prompt_version`, `structured_output` (JSONB), `narrative`

## 4. Index & Performance Strategy
- **RLS Performance**: Every query includes `WHERE organization_id = ...`. We will use **B-Tree indexes on `organization_id`** across all tables.
- **Time-Series**: **B-Tree on `created_at` or `occurred_at`** for trade history and scoring trends.
- **Search**: **GIN indexes** on JSONB columns (`rules`, `payload`) to allow complex filtering of strategy rules.
- **Unique Constraints**: Composite unique index on `(broker_account_id, external_id)` in `trade_events` to prevent duplicate imports.

---

## 5. Row Level Security (RLS) Strategy
- **Isolation**: Mandatory `organization_id` on all entities.
- **Functions**: A helper function `get_my_organizations()` returns UUIDs the user has access to.
- **Policies**: 
    - `SELECT`: `WHERE organization_id IN (get_my_organizations())`
    - `INSERT/UPDATE`: `WHERE organization_id IN (get_my_organizations('editor'))`

---

## 6. Storage Strategy
- **Bucket `trade-attachments`**:
    - Folder structure: `/{org_id}/{trade_id}/{type}_{uuid}.{ext}`
    - Access: Secured via Supabase Storage policies matching table RLS.
